## Purpose

Adapt Kronos-small to the 51-ticker Thai-retail universe by fine-tuning on 10 years of cached daily OHLCV. Produces a local checkpoint loadable via `KronosTH.from_checkpoint()`. Uses a 3-fold walk-forward split so the improvement over zero-shot is measured without look-ahead.

Approach: **full fine-tune** (all weights updated), two phases (tokenizer → predictor), fp16, aggressive checkpointing for Colab resilience.

## Rolling Fold Structure

3 folds, 6-month step. Each fold has disjoint val and test windows with no overlap with training data.

| Fold | Train end | Validation | Test |
|---|---|---|---|
| 0 | 2022-06-30 | 2022-07-01 – 2022-12-31 | 2023-01-01 – 2023-06-30 |
| 1 | 2022-12-31 | 2023-01-01 – 2023-06-30 | 2023-07-01 – 2023-12-31 |
| 2 | 2023-06-30 | 2023-07-01 – 2023-12-31 | 2024-01-01 – 2024-06-30 |

Training data for each fold: all cached history for each ticker up to (and including) `train_end`. Tickers with fewer than `lookback + pred_len` rows up to `train_end` are excluded from that fold silently.

Samples are generated by a sliding window (stride=1) over each ticker's training period:
```
for i in range(0, len(ticker_df) - lookback - pred_len + 1):
    x = ticker_df.iloc[i : i + lookback][ohlcva_cols]
    # Predict log-returns, not price levels (Issue #11)
    # Rationale: predicting absolute prices across BTC-USD ($100k) and PTT.BK (50 THB)
    # on the same loss scale means BTC gradients dominate by ~2000×. Log-returns are
    # dimensionless and comparable across all asset classes.
    close_window = ticker_df.iloc[i + lookback - 1 : i + lookback + pred_len]["close"]
    y = np.log(close_window.values[1:] / close_window.values[:-1])  # shape: (pred_len,)
    yield (x, y)
```
Val/test splits use `stride=pred_len` (non-overlapping windows) over their respective date ranges.

## Public API (`kth/models/finetune.py`)

```python
def prepare_dataset(
    cache_dir: str = "./data/raw",
    fold: int = 0,                       # 0, 1, or 2
    n_folds: int = 3,
    fold_step_months: int = 6,
    lookback: int = 400,
    pred_len: int = 20,
    class_weights: dict[str, float] | None = None,  # per-class sample weight (Issue #13)
    seed: int = 42,                      # reproducibility (Issue #14)
    streaming: bool = True,              # True → KronosDataset (lazy); False → list (eager) (Issue #15)
) -> dict[str, "KronosDataset"]:
    """
    Returns {"train": KronosDataset, "val": KronosDataset, "test": KronosDataset}.
    Each KronosDataset wraps (x_df, y_series) pairs:
      x_df:    shape (lookback, 6) — columns [open, high, low, close, volume, amount]
      y_series: shape (pred_len,)  — log-returns for the forecast horizon
    Train split: sliding window stride=1 over all history up to fold train_end.
    Val/test splits: sliding window stride=pred_len (non-overlapping windows) over
                     their respective date ranges, to avoid data leakage in evaluation.

    Always returns KronosDataset instances (not raw generators or lists).
    KronosDataset supports multiple iterations — finetune_tokenizer and
    finetune_predictor can both consume the same dataset dict without exhausting it.

    streaming=True (default): windows are built lazily per-ticker and appended to a list
      one ticker at a time before wrapping in KronosDataset. This keeps peak memory to
      one ticker's windows at a time during construction, avoiding a spike from
      allocating all tickers' windows simultaneously.
      Note: the final KronosDataset is still list-backed (~4-6 GB for full 51-ticker
      training set); streaming here refers to construction order, not lazy iteration.
    streaming=False: identical behaviour; parameter reserved for future lazy-file Dataset.

    class_weights: optional per-asset-class upsampling factors. If provided,
      samples from underweighted classes are duplicated to balance the loss.
      Example: {"thai_equity": 3.0, "crypto": 0.5} — Thai equities get 3x samples.

    seed: passed to numpy and torch for reproducible dataset shuffling and splits.
      Stored in training_args.json for audit.
    """


def finetune_tokenizer(
    dataset: dict[str, "KronosDataset"],
    output_dir: str,
    epochs: int = 1,
    batch_size: int = 32,
    lr: float = 1e-4,
    seed: int = 42,                      # reproducibility (Issue #14)
) -> str:
    """
    Fine-tune the Kronos VQ tokenizer on the training split.
    Returns output_dir (path) for use by finetune_predictor.
    Saves a HuggingFace-compatible checkpoint (config.json + weights).
    """


def finetune_predictor(
    dataset: dict[str, "KronosDataset"],  # same dict returned by prepare_dataset()
    tokenizer_path: str,
    output_dir: str,
    epochs: int = 5,
    batch_size: int = 8,
    grad_accum: int = 4,                  # effective batch = 32
    fp16: bool = True,
    lr: float = 5e-5,
    lr_scheduler: str = "cosine",         # "cosine" | "linear" | "constant" (Issue #4)
    weight_decay: float = 0.01,           # L2 regularisation (Issue #4)
    loss: str = "mae",                    # "mae" | "mse" | "huber" (Issue #4)
    early_stopping_patience: int = 2,     # stop if val loss doesn't improve for N epochs (Issue #4)
    save_every_n_steps: int = 200,        # checkpoint frequency for Colab resilience
    seed: int = 42,                       # reproducibility
) -> str:
    """
    Fine-tune the Kronos predictor (transformer backbone).
    Returns output_dir (path of best checkpoint by val loss). Loadable via KronosTH.from_checkpoint().

    seed: sets torch.manual_seed(seed), np.random.seed(seed), random.seed(seed)
          before any random operations. Ensures reproducible weight initialization
          and data shuffling across runs.

    Training notes (Issue #4):
    - lr_scheduler="cosine": cosine annealing from lr to lr/10 over all epochs with linear warmup
      for the first 5% of steps. Prevents overshooting minima on fine-tuning.
    - early_stopping_patience=2: training stops if val loss does not improve for 2 consecutive
      epochs. Saves the best checkpoint (lowest val loss), not the final epoch.
    - loss="mae": MAE on log-returns. MSE would over-penalise large moves (common in crypto),
      biasing the model toward predicting mean reversion. Huber is a middle ground.
    - weight_decay=0.01: applied via AdamW to all non-bias, non-LayerNorm parameters.
    """
```

### Checkpoint format
Both functions write a HuggingFace-compatible directory:
```
output_dir/
├── config.json
├── pytorch_model.bin   (or model.safetensors)
└── training_args.json  (lr, epochs, fold, seed, timestamp — for reproducibility)
```
This is the format expected by `KronosTH.from_checkpoint(output_dir)`.

### Regime caveat
The 3-fold structure covers 2022-2024 (post-COVID recovery, 2022 bear market, 2023-24 bull). These are three different regimes within a compressed 2.5-year window. A fine-tune that works across all three folds has survived a bear, a recovery, and a bull — but it has NOT seen a 2018 trade-war or 2020 COVID-crash regime. Every report and notebook MUST disclose this: the fine-tune is validated on 30 months, not a full market cycle. Extending folds back to 2018 (if data exists) would strengthen the claim considerably.

### `KronosDataset` class
Must be a **module-level** class (not defined inside a function). Defining it inside `_build_torch_dataset()` creates a dynamically-scoped class that cannot be pickled — PyTorch `DataLoader` with `num_workers > 0` would fail with `AttributeError: Can't pickle local object`. Define `KronosDataset(torch.utils.data.Dataset)` at the top level of `finetune.py`.

`KronosDataset` is the **return type of `prepare_dataset()`** and the **accepted type of `finetune_tokenizer()` / `finetune_predictor()`**. It wraps either an in-memory list or a lazy generator depending on `streaming`. Because it is a proper `Dataset` (not a raw generator), iterating over it multiple times is safe — `finetune_tokenizer` and `finetune_predictor` can consume the same dataset dict in sequence without the first call exhausting the data for the second.

### Empty validation set detection
Before calling `finetune_predictor`, check that `dataset["val"]` is non-empty. If all tickers had zero rows in the 6-month validation window (possible for earliest fold with recently-listed tickers), print a warning: `"Validation set is empty for fold {fold}. Early stopping disabled."` and proceed without val data.

## Checkpointing strategy

`save_every_n_steps=200` writes a checkpoint mid-epoch. If Colab disconnects:
- Each step checkpoint is a full model state (not a partial gradient state), so `from_checkpoint()` can load it cleanly
- Resume: load the latest step checkpoint via `KronosTH.from_checkpoint(step_ckpt_dir)`, extract the underlying `KronosPredictor`, and re-call `finetune_predictor()` with `epochs` reduced by the number already completed
- Google Drive mount (via notebook cell) persists checkpoints across sessions
