# Kaggle Scheduled Pipeline — Design Spec

**Date:** 2026-06-18
**Status:** Approved (scope decided 2026-06-18; revised v2 after senior review)
**Supersedes runtime in:** [2026-06-04-google-suite-dashboard-design.md](2026-06-04-google-suite-dashboard-design.md) (Colab T4 → Kaggle)

---

## Problem

The daily pipeline runs on **Google Colab**, which has no API and no unattended
scheduler on the free tier — a human must open the notebook and run cells. We want a
**$0, GPU-backed, scheduled** runtime that updates the Google Suite dashboard
automatically.

## Decision summary

| Decision | Choice | Why |
|---|---|---|
| Runtime | **Kaggle Notebooks** (free P100/T4 16GB) | Only free GPU with a built-in scheduler |
| Trigger | **Scheduled only** (Kaggle native daily schedule) | $0, no extra moving parts |
| On-demand "Run now" | **Out of scope** | Needs GitHub Actions = CI (hard scope limit) |
| Colab notebook | **Retired — but only after 2 green scheduled days** | Keep rollback + dev surface until proven |
| Auth to Sheets | **Service account** (Kaggle Secret) | Unattended runs can't do interactive OAuth |
| Data store / bridge | **Google Sheets (unchanged)** | Apps Script dashboard already reads Sheets |
| Execution model | **At-least-once / idempotent** | Schedulers retry; same-day re-runs must not double-write |
| Clock | **`Asia/Bangkok`, injected** | `date.today()` is UTC on Kaggle → wrong forecast dates |
| Tests | **Plain-assert `verify_*.py`** (no pytest/CI) | Project hard-scope-limit; mirrors `verify_data_layer.py` |

## Architecture

```
Kaggle Notebook  (scheduled daily, GPU, Internet ON)
  └─ kth.pipeline.daily.run_daily_pipeline(gc, spreadsheet_id, model=…, data_loader=…,
                                           today=<BKK date>, notifier=…)
        Capital Reset (4b) ─► state ─► Trade Edits (9b) ─► forecasts ─► fills ─►
        ticket ─► metrics ─► staging→promote (idempotent upserts) ─► Pipeline Status
Google Sheets  (source of truth — 18 tabs, unchanged)
        ▲
        │ reads (60s cache, SPA auto-refresh)
Apps Script Web App  ──►  dashboard (unchanged)
```

The Apps Script / dashboard side does **not** change. New data appears within ~60s
(existing auto-refresh).

## What changes vs. Colab

1. **Auth.** `google.colab.auth.authenticate_user()` + `google.auth.default()` →
   **service account**: `gspread.service_account_from_dict(sa_info)`. Secret `GCP_SA_JSON`
   in Kaggle Secrets; spreadsheet shared with the SA email (**Editor**).
2. **Inputs.** `userdata.get(...)` → `UserSecretsClient().get_secret(...)`, abstracted behind an
   injected `getter` so tests use a fake and local runs use env vars. Secrets: `GCP_SA_JSON`,
   `SPREADSHEET_ID`, `HF_TOKEN` (optional), `GITHUB_PAT` (if repo is private).
3. **State.** Kaggle FS is **ephemeral** — no Drive. The JSON-bridge rebuilds local JSON from
   Sheets each run, so **Sheets stays the source of truth**; local JSON is throwaway under a
   temp dir.
4. **Queued mutations now run in the scheduled flow.** Capital Reset (4b) and Trade Edits (9b)
   were stand-alone "run-when-banner-shows" cells. The scheduled run **must** apply them, in
   order: **4b before forecasts** (first-run SETUP must take effect), **9b after state rebuild
   and before ticket generation**.
5. **Idempotency.** Equity Curve and Forecast History are **upserted by date key** (replace
   today's row, never blind-append) so a retry or same-day re-run is safe.
6. **Clock.** `today` is computed as `datetime.now(ZoneInfo("Asia/Bangkok")).date()` and passed
   in; all `date.today()` usage inside the lifted logic is replaced by this injected value.
7. **Alerting.** On failure the run writes `status=failed` to Pipeline Status **and** calls an
   injected `notifier(level, msg)` (email/webhook) so a silent failure can't hide.
8. **Repo + deps.** Pinned-commit clone (`GITHUB_PAT` if private) → `pip install -e .`. Internet
   ON (yfinance, HF, gspread). **Model weights cached as a Kaggle Dataset by default** (cold HF
   download is the fallback, not the norm, for a daily job).
9. **Notebook.** A thin wrapper calling `run_daily_pipeline`; generated by a builder.

## Risk register (ordered)

| # | Risk | Severity | Mitigation | Verified in |
|---|---|---|---|---|
| R1 | Kaggle **scheduled** runs may not grant GPU / Internet / Secrets together | **Blocker** | Phase 0 spike proves the full trifecta on a schedule before any refactor | Phase 0 |
| R2 | **yfinance blocked/rate-limited** from Kaggle shared IPs | **High** | Spike tests a real download; contingency: pre-cached OHLCV Kaggle Dataset + incremental update, or alternate source | Phase 0 |
| R3 | Same-day retry **double-writes** equity/forecast history | High | Date-keyed upsert; idempotency asserted offline | Phase 2 |
| R4 | `date.today()` UTC → **wrong forecast_date** | High | Inject BKK `today` | Phase 2 |
| R5 | Dashboard edits/setup have **~24h latency** (no on-demand trigger) | Medium (UX) | Documented trade-off; scheduled run applies 4b/9b every run | accepted |
| R6 | Secrets unavailable / size limits in scheduled context | Medium | Spike confirms; base64-encode SA JSON; `load_secrets` handles both | Phase 0/1 |
| R7 | Silent unattended failure → stale dashboard | Medium | `notifier` on failure + Pipeline Status banner | Phase 2/5 |
| R8 | Sheets API write quota / run time on full rewrite | Low | `write_staging` already batches per tab; keep inter-tab sleeps | Phase 5 |
| R9 | Cold model download slow/flaky daily | Low | Weights Kaggle Dataset default | Phase 3 |

**Hard gate:** if Phase 0 cannot prove R1 and R2, **stop** — Kaggle is not viable and we
fall back to local-GPU cron (`scripts/dashboard.py --generate`); do not invest in the refactor.

## Accepted trade-offs (explicit)

- **No on-demand trigger** → dashboard-initiated changes (set capital, edit/delete trade) apply
  on the **next scheduled run** (up to ~24h). This is the cost of dropping CI/GitHub Actions.
- **Retiring Colab** removes the interactive runtime; mitigated by keeping it until Phase 5 is
  green and by the `--dry-run` offline path for development.

## Out of scope

On-demand "Run now" button (CI), live order execution, intraday data (existing hard limits),
keeping Colab as a permanent parallel runtime.
